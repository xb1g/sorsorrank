# Engineering Review: Static Mobile Architecture

Generated by `/plan-eng-review` on 2026-05-21.

## Verdict

Yes, this is possible, but not as a pure static-only app.

The correct architecture is:

```text
Static mobile frontend
  -> small serverless/BaaS API
  -> database for roster, aggregates, consent, and short-retention events
```

The frontend can be fully static and hosted on a CDN. The dynamic parts need a backend because daily limits, ranking aggregates, abuse prevention, retention cleanup, takedown intake, and admin controls cannot be trusted to client-side code.

## Recommended Stack

### Frontend

- Vite static build.
- Preact or vanilla TypeScript.
- Plain CSS/CSS modules.
- Web Animations API, CSS transitions, or one small motion helper.
- No Next.js, SSR, server components, or heavyweight routing.

### Backend

Recommended default: Supabase.

- Supabase Edge Functions for mutation APIs.
- Postgres for roster, consent, swipe events, aggregates, contact requests, and audit logs.
- Row Level Security enabled on all tables.
- Server-side functions for swipe recording and aggregation.

Alternative: Cloudflare Pages + Functions + D1/KV.

- Good if we want one edge/CDN vendor.
- Slightly more custom plumbing for relational aggregates and admin flows.

## Why Not Pure Static?

Pure static can do:

- Render the swipe UI.
- Store local-only demo state.
- Link to Google.
- Generate a local share image.

Pure static cannot safely do:

- Real daily limits.
- Abuse-resistant swipe recording.
- Aggregate public rankings.
- Privacy-safe retention cleanup.
- Admin roster updates.
- Takedown/contact workflow.
- Feature flags/freeze mode.
- Server-generated social previews.

## Data Flow

```text
visitor opens static app
  -> app loads roster/deck from API
  -> consent accepted
  -> user swipes Research/Skip
  -> API validates consent + daily limit + idempotency
  -> API writes short-retention event
  -> API updates daily aggregate
  -> app shows next animated card
  -> after 10, app shows completion share card
  -> rankings read aggregate only
```

Shadow paths:

- No consent: API rejects swipe; app returns to consent gate.
- Daily limit reached: API rejects 11th swipe; app shows done state.
- Duplicate tap: idempotency key no-ops the second request.
- DB failure: API returns retry-safe error; app does not optimistically double count.
- Offline: app shows retry state; no silent local queue for political actions unless explicitly designed.

## Mobile Animation Plan

Animations should be fun but cheap:

- Drag card with finger.
- Slight tilt while dragging.
- Snap card away on decision.
- Progress tick from `1/10` to `10/10`.
- Completion burst after the 10th card.
- Share-card preview slides up.

Performance rules:

- Animate `transform` and `opacity`, not layout-heavy properties.
- Keep card DOM small.
- Preload the next card data.
- Respect `prefers-reduced-motion`.
- Avoid large animation libraries.

## Error & Rescue Registry

| Codepath | Failure | Rescue | User sees |
|---|---|---|---|
| `accept-consent` | missing/invalid consent version | reject and return current version | consent retry |
| `get-deck` | no active roster | return empty deck state | no cards available |
| `get-deck` | freeze mode active | return frozen state | rankings paused |
| `record-swipe` | no consent | reject | consent required |
| `record-swipe` | 11th swipe | reject without write | done for today |
| `record-swipe` | duplicate idempotency key | return previous success | no duplicate count |
| `record-swipe` | invalid politician id | reject and log | card unavailable |
| `record-swipe` | aggregate write fails | rollback event and return retryable error | try again |
| `get-rankings` | below sample threshold | hide rank row | insufficient data |
| `create-share` | share disabled | return disabled state | share unavailable |
| `submit-contact` | spam/rate limit | throttle | try later |
| `admin-roster` | unauthorized | deny and audit | forbidden |

Critical gap to avoid: direct client writes to swipe or aggregate tables.

## Security Review

Main risks:

- Political-opinion data stored too long.
- Client writes forged swipe events.
- Bot traffic manipulates rank.
- Roster text injects script.
- Share cards imply endorsement.
- Admin controls exposed.

Required mitigations:

- Server-side swipe recording.
- Consent check in the serverless function.
- Idempotency key per card impression.
- Daily limit enforced server-side.
- RLS enabled on database tables.
- Admin APIs authenticated.
- XSS escaping for roster text.
- Short raw-event retention.
- Copy regression tests for banned terms.

## Test Plan

### Frontend Tests

- Card drag and button fallback both work.
- `prefers-reduced-motion` disables major motion.
- Long Thai/English names fit.
- Offline swipe shows retry state.
- Completion card appears after 10.
- Already-done state appears on repeat visit.

### API Tests

- Cannot record swipe without consent.
- 10th swipe succeeds, 11th fails.
- Duplicate idempotency key does not double count.
- Research increments positive aggregate.
- Skip increments impression only.
- Invalid politician id is rejected.
- Freeze mode pauses deck/ranking behavior.

### Security Tests

- XSS roster values render as text.
- Direct database write is impossible from public client.
- Unauthorized admin calls fail.
- Rate limit rejects suspicious bursts.
- Raw event cleanup removes old events.

## Performance Review

Frontend:

- Static assets should be CDN-cached.
- Initial bundle should stay small.
- Motion should use `transform`/`opacity`.
- Avoid image-heavy politician cards unless rights and optimization are solved.

Backend:

- Ranking page should read precomputed aggregates, not scan raw events.
- Swipe path should be one short transaction.
- Add indexes on visitor/day, politician/day, and aggregate date.
- Retention cleanup should run on schedule.

## Deployment Plan

```text
1. Deploy static frontend shell.
2. Deploy database schema with RLS.
3. Deploy serverless functions.
4. Seed small roster.
5. Enable private alpha.
6. Verify retention cleanup and aggregate correctness.
7. Enable share cards.
8. Enable public rankings after sample threshold.
```

Rollback:

```text
turn off swipe_enabled
  -> hide rankings_public
  -> keep methodology/contact available
  -> preserve audit logs
  -> revert static bundle if needed
```

## Decision

Build static-first. Do not use Next.js unless a future requirement genuinely needs SSR or server-rendered social previews. For v1, use Vite + Preact or vanilla TypeScript, with Supabase Edge Functions/Postgres as the dynamic layer.
